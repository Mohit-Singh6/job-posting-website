## Why are there both `jwt` and `session` properties? (Aren't they different?)

You are right that in general web development, **JWT** (stateless tokens) and **Sessions** (stateful database tracking) are different strategies. However, inside Next-Auth, they work together as a nested system.

When you set `strategy: 'jwt'`, Next-Auth uses a **JWT-based Session** architecture:

* **The JWT:** This is the actual **encrypted cookie** container (`httpOnly`) stored in the browser. The frontend JavaScript cannot read this directly because it is locked down securely on the server.
* **The Session:** This is the **unencrypted data object** that Next-Auth securely decrypts and exposes to your frontend components (via hooks like `useSession()` or server functions like `auth()`).

Think of the **JWT** as a locked safe traveling between the browser and server, and the **Session** as the specific items you choose to take *out* of that safe to show on the user's screen.

## What is the use of these callbacks? (What do they do?)

Next-Auth passes data down like an assembly line. By default, it hides the user's database `id` from the frontend for security reasons. You use these two callbacks to intentionally pass that `id` down the pipeline.

```
[GitHub/Prisma DB User Profile]
             │
             ▼
   ┌───────────────────┐
   │   jwt() Callback  │  ◄── 1. Intercepts the DB User object and
   └─────────┬─────────┘         manually saves "user.id" onto "token.id"
             │
             ▼
   ┌───────────────────┐
   │ session() Callback│  ◄── 2. Takes the saved "token.id" and explicitly
   └─────────┬─────────┘         injects it into the public "session.user.id"
             │
             ▼
[Available in App Components via auth()]

```

### 1. The `jwt` Callback (Server-Side Storage)

This function runs whenever a JWT is created or updated. It has direct access to the `user` record from your Prisma database.

* **What it does:** It allows you to inject properties (like `user.id`) into the encrypted cookie layout (`token`). Once saved here, the data is securely stored but still hidden from your React components.

### 2. The `session` Callback (Client/UI Exposure)

This function runs whenever your application requests the session state (like calling `auth()` or `useSession()`). It receives the `token` generated by the step above.

* **What it does:** By default, Next-Auth only exposes a user's `name`, `email`, and `image` to the browser. This callback acts as a gateway where you explicitly map `session.user.id = token.id`. This makes the user ID safely accessible inside your pages and forms.

I get the confusion! Let's clear this up completely.

Yes, in traditional web development (like Node/Express, PHP, or Python), you **do** use just *one* thing: a traditional Database Session.

Next-Auth uses both because **Next.js runs on Serverless/Edge hosting** (like Vercel). Traditional single-thing sessions break in this environment.

Here is the dead-simple difference:

---

### 1. Traditional Single-Thing Session (Express / PHP)

You have one giant, permanent server running 24/7 with a database attached.

```
[Browser Cookie: "ID: 123"] ──> [One Big Persistent Server] ──> Looks up "123" in Database

```

* **How it works:** The browser cookie holds a tiny ID number (e.g., `123`). The server sees `123`, looks at its local memory or database, finds the user, and serves the page.
* **Why it's "one thing":** The server never changes, so tracking is simple.

---

### 2. The Next.js Problem (Why we need two things)

Next.js doesn't run on one big persistent server. It runs on **Serverless Functions**. When a user clicks a page, a tiny server boots up instantly in milliseconds, serves the page, and **immediately dies**.

Because the server disappears, it cannot hold a constant connection to a database to look up session ID `123` on every single click. It would be incredibly slow and crash your database.

### How Next-Auth solves this with "Both Things"

To survive in a serverless world, Next-Auth uses the two-step setup:

1. **The JWT:** Instead of looking up a user in a database on every click, the server encrypts the *entire user profile* directly into the cookie (the JWT). The dying/rebooting servers can instantly decrypt this cookie without touching a database.
2. **The Session:** Because that cookie is securely encrypted so hackers can't read it, Next-Auth converts it into a plain JavaScript object (the Session) for one split second so your React frontend can read the user's name.

**Summary:** You use two things because Next.js is serverless. The JWT replaces the permanent server memory, and the Session lets React see the data.
